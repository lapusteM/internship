$. Responsive design principles and best practices




Think twice, code once!

Radu B. Gaspar

The basic principle behind responsive design is one site for every screen.


When to use it? Things to consider: time & money, browser support, performance, content, website or web app


How to do it? One way is to use frameworks which save time; another way is to create your own system which gives you more control in the detriment of time.


Best practices? Content check, start small (mobile first), make it modular, always optimize.

There are two main approaches:

Adaptive: which consist of multiple fixed width layouts, and

Responsive: which consist of multiple fluid grid layouts

The mixed approach would be using fixed width for large and medium screens and fluid width for small screens.


Our end goal is to have one site for evey screen... so what's the challenge?


iOS Screen Fragmentation

Android Screen Fragmentation


We're still just taking about only 2 companies that provide smart devices... but those aren't the only ones, we still have Windows Phone, BlackBerry and others.

To make matters worse, not all of them have the same browser, and implicitly not the same browser feature support.


Seamingly simple things like fixed positioning are not possible on Windows Phone 7 or Android <= 2.3 devices. The only way to achieve a similar effect is using JavaScript.


We still haven't said a word about web browsers, each with their own unique quirks. Mobile web browsers and Desktop web browsers don't behave the same, not even those created by the same company!


The reason for that is simple, a mobile device has a more finite amount of resources when compared to desktops in general.

So which is the best framework? Depends on the project.


Whenever starting a project, the first question(s) should be: Does it need to be responsive? and What browser support are you expecting?


Based on the answers you might need to take legacy browsers into account.


Bootstrap seems to cover a lot of the needs and issues that developers encounter today and unlike other frameworks it uses both CSS and JavaScript.


Bootstrap home page.

What's the key behind responsive layouts? Relative width containers (flexible width) and CSS media queries.


What browsers support them? See "CanIUse.com"


What about the rest? Good old-fashioned JavaScript. Fully featured frameworks like ExtJS use JavaScript, at their core, for component rescaling. Anything and everything you use is scaled based on your browsers size.


It should be mentioned at this stage that simply rescaling content does not make your site responsive!


Responsiveness also encompasses the look and feel of the site. You also have to reposition the contents of the site in addition to rescaling it to fully take advantage of the screen you have to work with.

The size and layout of the content is different on all screens. This is what we mean by responsive.

Talk about doing more than one thing at once...


...that is what responsive sites are really about.


It will add to your development time and it's not a walk in the park even using frameworks.


Testing on multiple devices is mandatory and development time will at least be doubled based on how many devices need to be supported.

What you will need is a list of displays by pixel density, luckily somebody already thought of that


Alternatively you could use a simplified version when working towards responsive design.


Make it easy. Use JavaScript to detect the browser width (and/or height as needed), then apply a CSS class on the body that restyles the content based on that class.


var ww = $(window).width(),
    body = $("body").removeClass();
if(ww > 1200) {
    body.addClass("large");
} else if(ww >= 800 && ww <= 1200) {
    body.addClass("medium");
} else {
    body.addClass("small");
}

Now we can use CSS to use those classes:


body.large .content { width: 1500px; }
body.medium .content { width: 900px; }
body.small .content { width: 600px; }

These classes and styles can be as fine-grained as you need them and using classes in apropriate places can trigger cascading layout changes.


Use CSS inheritance to make things easier.


Using multiple HTML layouts for the same thing is not be required. If you need a menu to look different depending on the screen size, that change can be handled by CSS and a bit of JavaScript.

Bootstrap does this with it's navigation bars, which toggle from an inline menu to a simple button.


That button has an event attached to it which displays a dropdown menu with your navigation items.


That's pretty easy to achieve and here's how:


/* styles that apply on large screens */
body.large .menu-inline { /* display the larger inline menu */ }
body.large .menu-collapsed { display: none; /* hide the collapsed menu */ }

/* styles that apply on small screens */
body.small .menu-inline { display: none; /* hide the larger menu */ }
body.small .menu-collapsed { /* display the collapsed menu */ }

By changing a class on the body (or a container of our choice, like a menu), we can affect all of our DOM elements using simple CSS rules.

The Boostrap menu is no different. As soon as your browsers width drops beneath a certain value, a class is added to the menu which restyles the entire thing.


Responsive design isn't the easiest thing in the world, and it's definitely not the hardest but the alternative is multiple sites which are device specific.
